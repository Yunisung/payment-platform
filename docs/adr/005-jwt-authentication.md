# ADR 005 - JWT 기반 인증 방식 선택

## 상태
채택 (Accepted)

## 배경

결제 API는 가맹점(MERCHANT)과 어드민(ADMIN) 두 가지 역할을 구분해야 합니다.
이를 위한 인증 방식으로 **서버 사이드 세션**과 **JWT** 중 하나를 선택해야 했습니다.

## 결정

**JWT (JSON Web Token)** 기반 인증 선택

- HMAC-SHA256으로 서명
- claim에 `memberId`, `role` 포함
- 만료 시간 설정 (application.yml의 `jwt.expiration`)
- Refresh Token 미구현

## 비교

| 항목 | 서버 사이드 세션 | JWT |
|------|----------------|-----|
| 상태 저장 위치 | 서버 메모리 또는 Redis | 클라이언트 (토큰 자체) |
| 수평 확장 | 세션 공유 문제 발생 | 서버 무상태 — 어느 인스턴스에서도 검증 가능 |
| 로그아웃 처리 | 서버에서 즉시 무효화 | 토큰 만료 전까지 유효 (블랙리스트 필요) |
| 네트워크 비용 | 세션 ID만 전송 | 토큰 크기만큼 매 요청 전송 |
| 권한 정보 | 서버 조회 필요 | 토큰 내 claim에서 즉시 추출 |

## 이유

### JWT를 선택한 핵심 이유

**1. 수평 확장 시 세션 공유 문제 없음**

서버 사이드 세션은 api-server를 2대 이상으로 늘릴 때
세션 저장소를 Redis로 별도 구성하거나 Sticky Session을 설정해야 합니다.
JWT는 서버가 무상태(Stateless)이므로 어느 인스턴스가 요청을 받아도
시크릿 키만 공유되면 독립적으로 검증할 수 있습니다.

**2. 역할(Role) 정보를 토큰에 내장**

MERCHANT/ADMIN 구분을 매 요청마다 DB를 조회하지 않고
토큰 claim에서 바로 추출해 사용합니다.

```
토큰 claim
{
  "sub": "1",          // memberId
  "role": "MERCHANT",  // 권한
  "exp": 1700000000
}
```

## 트레이드오프

### Refresh Token을 구현하지 않은 이유

Refresh Token을 구현하면 서버에 토큰 저장소(Redis 등)가 필요하고,
로직 복잡도가 크게 올라갑니다.

이 프로젝트의 목적은 결제/정산 도메인 핵심 로직 구현에 있었기 때문에,
Access Token만 발급하고 만료 시 재로그인하는 단순한 방식을 선택했습니다.

실서비스에서는 토큰 탈취 시 즉시 무효화가 불가능하다는 JWT의 단점을 고려해
Refresh Token + 블랙리스트 방식이 필요합니다.

| 항목 | 내용 |
|------|------|
| 장점 | 무상태 서버, 수평 확장 용이, DB 조회 없이 권한 확인 |
| 단점 | 토큰 만료 전 강제 무효화 불가, Refresh Token 미구현으로 UX 제한 |
| 보완 | 만료 시간을 짧게 유지해 탈취 피해 최소화 |
