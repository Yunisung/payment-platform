# ADR 003 - 멀티모듈 구조 설계

## 상태
채택 (Accepted)

## 배경

결제 플랫폼은 세 가지 역할이 명확히 구분됩니다.

- **api-server**: 클라이언트 요청 처리 (동기)
- **settlement-service**: 이벤트 기반 정산 처리 (비동기)
- **batch**: 주기적 정산 집계 (스케줄)

이를 단일 모듈로 구성할지, 멀티모듈로 분리할지 결정이 필요했습니다.

## 결정

**Gradle 멀티모듈 프로젝트**로 구성하되,
각 모듈을 **독립적으로 배포 가능한 Spring Boot 애플리케이션**으로 유지

## 이유

### 단일 모듈의 문제

```
단일 모듈로 구성 시
  api-server 배포 → 전체 재시작
  batch 스케줄 변경 → 전체 재배포
  settlement 부하 증가 → 전체 스케일아웃 (낭비)
```

### 멀티모듈의 장점

**1. 독립 배포**

API 트래픽이 급증해도 api-server만 스케일아웃하면 됩니다.
batch나 settlement-service는 영향을 받지 않습니다.

**2. 장애 격리**

settlement-service가 다운되어도 결제 API는 정상 동작합니다.
Kafka에 이벤트가 쌓이고, 복구 후 순차 처리됩니다.

**3. 기술 스택 유연성**

각 모듈이 독립적이므로 필요 시 다른 언어/프레임워크로 교체 가능합니다.

### 모듈 간 코드 공유 없이 설계한 이유

공통 모듈(`common`)을 만들어 엔티티를 공유하는 방식도 있지만,
이 프로젝트에서는 의도적으로 **각 모듈이 자체 도메인 모델을 소유**하도록 했습니다.

- settlement-service의 `Settlement`와 batch의 `Settlement`는 같은 테이블을 바라보지만
  각자의 관심사(이벤트 수신 vs 배치 처리)에 맞게 독립적으로 정의
- 공통 모듈 의존성이 생기면 한 모듈 변경이 전체에 영향을 미치는 결합도 증가

## 트레이드오프

| 항목 | 내용 |
|------|------|
| 장점 | 독립 배포, 장애 격리, 모듈별 스케일아웃 |
| 단점 | 유사한 코드 중복 (Settlement 엔티티 등), 초기 설정 복잡 |
| 보완 | 중복은 의도적 선택 — 결합도 최소화 우선 |
