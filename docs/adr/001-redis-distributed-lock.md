# ADR 001 - 멱등성 보장을 위한 Redis 분산락 선택

## 상태
채택 (Accepted)

## 배경

결제 API는 네트워크 오류나 클라이언트 재시도로 인해 동일한 요청이 여러 번 들어올 수 있습니다.
이를 처리하지 않으면 중복 결제가 발생하므로, **멱등성(Idempotency)** 보장이 필수입니다.

고려한 방식은 크게 두 가지였습니다.

| 방식 | 설명 |
|------|------|
| DB Unique Index | idempotencyKey 컬럼에 Unique 제약 추가 |
| Redis 분산락 | 요청 처리 전 Redis에 락을 획득 |

## 결정

**Redis 분산락 + DB Unique Index 이중 적용**

- 1차 방어: Redis 분산락으로 동시 요청 차단
- 2차 방어: DB Unique Index로 데이터 정합성 보장

## 이유

### DB Unique Index만 사용할 경우의 문제

동시에 두 요청이 들어오면 둘 다 INSERT를 시도합니다.
하나는 성공하고 하나는 `DuplicateKeyException`이 발생하지만,
이미 두 트랜잭션이 열린 상태에서 DB에 불필요한 부하가 발생합니다.

### Redis 분산락의 장점

```
요청 A ──► Redis tryLock("key-001") → 성공 → 결제 처리
요청 B ──► Redis tryLock("key-001") → 실패 → 즉시 409 반환
```

- 락 획득 실패 시 DB까지 가지 않고 **즉시 차단**
- TTL(10초) 설정으로 락 미해제 시에도 자동 만료
- 수평 확장(다중 서버) 환경에서도 동작

### 이중 체크 패턴 적용

락 획득 후에도 DB를 재확인합니다.
락 획득 전에 이미 처리된 요청이 있을 수 있기 때문입니다.

```
요청 수신
  → DB 조회 (기존 결과 있으면 반환)
  → Redis 락 획득 시도
  → 락 성공 후 DB 재조회 (이중 체크)
  → 결제 처리
```

## 트레이드오프

| 항목 | 내용 |
|------|------|
| 장점 | 동시 중복 요청 조기 차단, 다중 서버 환경 지원 |
| 단점 | Redis 장애 시 락 획득 불가 (가용성 의존) |
| 보완 | DB Unique Index를 2차 방어선으로 유지 |
